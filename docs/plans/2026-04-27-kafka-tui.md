# kafka-tui — TUI client for Kafka

## Overview

Implementation of a TUI application for working with Kafka clusters in Go (Bubble Tea v2 + franz-go). Supports browsing/managing topics, messages, consumer groups, produce, with k9s-style navigation, configs with hierarchical merging, Vault, and read-only protection for prod clusters. Full functional specification in plan.md.

## Context

- Files involved: new code in `cmd/kafka-tui/`, `internal/config/`, `internal/kafka/`, `internal/tui/`, `internal/vault/`, `internal/clipboard/`, `internal/state/`, `internal/logging/`
- Related sources: `plan.md` (full feature specification), `Makefile`, `go.mod`, `mise.toml`, `.golangci.yml`
- External dependencies: `github.com/charmbracelet/bubbletea/v2`, `github.com/charmbracelet/lipgloss`, `github.com/twmb/franz-go`, `github.com/twmb/franz-go/pkg/kadm`, `github.com/hashicorp/vault/api`, `github.com/fsnotify/fsnotify`, `gopkg.in/yaml.v3`, `modernc.org/sqlite` (to avoid CGO), `github.com/spf13/pflag` (for CLI)
- Module path: `github.com/aleksey925/kafka-tui`

## Development Approach

- **Testing approach**: Regular (code first, tests after) — for TUI this is more reasonable since there are many visual components
- Each task is fully completed before moving on to the next
- Business logic (config, vault, kafka wrappers) is covered by unit tests; TUI — by golden tests via `teatest` where reasonable
- **CRITICAL: every task MUST contain new/updated tests**
- **CRITICAL: all tests must pass before starting the next task**
- After each task: `make lint && make test`
- Vault and Kafka in tests are spun up via testcontainers
